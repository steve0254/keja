# Priority 2 — Real AI Assistant: what changed

## Summary

Replaced the previous single-shot "ask Lovable's AI gateway for a JSON blob"
implementation with a real, provider-agnostic AI service layer:

- **`src/services/ai/`** — the service layer.
  - `providers/{openai,anthropic,gemini}.ts` — one adapter per provider,
    talking to each provider's real REST API directly (no SDK dependency).
    Picked via `AI_PROVIDER` env var, or auto-detected from whichever API
    key is set.
  - `tools/definitions.ts` + `tools/execute.ts` — real function-calling
    tools backed by actual Supabase queries: `search_listings`,
    `get_listing`, `compare_listings`, `estimate_commute`,
    `get_comparable_listings`. The model converts natural language into
    real filters and gets real data back — it never invents a listing.
  - `prompts.ts` — versioned system prompts per assistant mode (tenant
    search vs. landlord description vs. landlord price).
  - `conversation.ts` — history sanitizing/trimming.
  - `rate-limit.ts` + migration `20260722120000_ai_rate_limits.sql` — atomic,
    Postgres-backed rate limiting (in-memory counters don't work across
    Cloudflare's stateless edge instances).
  - `identity.server.ts` — resolves signed-in user vs. anonymous IP for
    rate-limit keying and landlord-tool auth gating.
  - `assistant.ts` — the orchestrator: a generic tool-calling loop (works
    the same regardless of provider) that streams the final answer token by
    token over Server-Sent Events.
- **`src/hooks/useAI.ts`** — client hook that drives the SSE stream and
  exposes `{ entries, isStreaming, send, reset }`. This is what makes the
  "typing" animation real — it's actual token streaming, not a fake timer.
- **`src/components/assistant/`** — `MessageBubble`, `ListingResultsStrip`,
  `LandlordAiPanel`.
- **`src/components/AiSearchChat.tsx`** — rebuilt on `useAI`, same UI shell.
- **`src/routes/add.tsx`** — landlord flow now has an inline AI helper that
  writes a description and suggests a price range grounded in comparable
  listings from Supabase.
- Removed `src/lib/ai-search.ts` (the old Lovable-gateway implementation).

## New dependencies

**None.** All three providers are called directly via `fetch`, matching the
existing code style and keeping the bundle lean.

## Environment variables required

See `.env.example` for the full list. New ones:

| Variable | Required | Notes |
|---|---|---|
| `AI_PROVIDER` | No | `openai` \| `anthropic` \| `gemini`. Auto-detected from whichever key below is set if omitted. |
| `OPENAI_API_KEY` / `OPENAI_MODEL` | If using OpenAI | Default model `gpt-4o-mini`. |
| `ANTHROPIC_API_KEY` / `ANTHROPIC_MODEL` | If using Anthropic | Default model `claude-sonnet-4-6`. |
| `GOOGLE_AI_API_KEY` / `GOOGLE_AI_MODEL` | If using Gemini | Default model `gemini-2.5-flash`. |
| `AI_RATE_LIMIT_PER_MINUTE` | No | Default `12`. |

All of these are server-only — verified after a production build that no
key or provider URL appears in the client bundle.

## Database changes

New migration `supabase/migrations/20260722120000_ai_rate_limits.sql`:
- `ai_rate_limits` table (RLS on, no public policies — only touched by the
  service-role client via a `SECURITY DEFINER` function).
- `check_ai_rate_limit(client_key, limit, window_seconds)` — atomic
  sliding-window check-and-record function, serialized per key with an
  advisory lock.

**Also patched `src/integrations/supabase/types.ts`** to add the
`check_ai_rate_limit` function signature, since it's normally
auto-generated by `supabase gen types` — run that command (or apply the
migration and regenerate) once this is linked to your live Supabase
project, to make sure the generated types stay authoritative.

## Model picks

Model names default to reasonably current, stable choices as of this
writing, but AI providers ship new models often — worth checking each
provider's docs before launch and adjusting `OPENAI_MODEL` /
`ANTHROPIC_MODEL` / `GOOGLE_AI_MODEL` if a newer/cheaper model fits better.

## Remaining recommendations before production launch

1. **Smoke-test each provider you plan to use** with a real API key —
   confirm tool-calling and streaming both work end to end.
2. **Apply the migration** to your Supabase project and run
   `supabase gen types` to make the generated types the source of truth
   again (they were hand-patched here to keep the build green).
3. **Tune `AI_RATE_LIMIT_PER_MINUTE`** for your expected traffic and budget
   — this bounds provider API spend as much as it prevents abuse.
4. **Decide on a cost/quality tradeoff per provider** — the assistant makes
   at least 2 model calls per user turn (a tool-decision call, then a
   streamed final-answer call; sometimes a 3rd for a second tool round).
   Cheaper/faster models are a reasonable default for the tenant search
   assistant; a stronger model may be worth it for landlord price
   suggestions.
5. **`estimate_commute` is a straight-line estimate**, not real routing —
   it's clearly labeled as such in both the tool description and the system
   prompt, but it should ideally be replaced by the real Directions API
   integration once Priority 3/4 (navigation) lands.
6. Consider **persisting conversation history** (currently in-memory per
   page load) if you want continuity across navigation — was left out to
   keep this change scoped, but `useAI` is structured so that's a small
   addition later.
